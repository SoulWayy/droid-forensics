Worker teardown (5 seconden!)

  1. SIGTERM ontvangen
  2. JsonRpc.shutdown() — buffer legen, pending requests cancel
  3. ProcessTracker.killAll() — tool processen doden
  4. McpHub.close() — MCP servers stoppen (linear, notion, playwright)
  5. CustomerOtelClient.flush() — telemetry wegschrijven
  6. CustomerOtelClient.shutdown()
  7. TuiDaemonAdapter.close()

De 5s teardown komt waarschijnlijk door open HTTP verbindingen (LLM streaming) die niet direct sluiten.

🔬 Stap-voor-stap dissectie

Stap 1: getReadinessHint in useMemo (FOUT)

Header.tsx gebruikt useMemo om de git readiness hint te berekenen. useMemo hoort pure te zijn (geen side-effects). Maar getReadinessHint roept statSync aan om .git te vinden in elke parent directory — dat is een synchronis I/O side-effect tijdens render.

// Hypothetische reconstructie wat Droid doet (FOUT):
const readiness = useMemo(() => {
  // ⚠️ SYNCHRONIS I/O TIJDENS RENDER ⚠️
  return getReadinessHint(cwd);
}, [cwd]);

// Wat het zou moeten zijn:
const [readiness, setReadiness] = useState(null);
useEffect(() => {
  getReadinessHintAsync(cwd).then(setReadiness);
}, [cwd]);

Stap 2: statSync door de hele tree

git.ts:18 zoekt .git in huidige dir, dan parent, dan parent's parent...

Bij elke render (die door elke notification triggert) wordt dit pad afgelopen:

statSync('/home/jan/utrecht-data-os/.git') → ENOENT
statSync('/home/jan/.git')                 → ENOENT
statSync('/home/.git')                     → ENOENT
statSync('/.git')                          → ENOENT

Dit zijn 4 sync syscalls per render. Bij 100 renders/sec = 400 statSync/sec → CPU overload.

Stap 3: Duplicate keys

De code genereert keys als header-${category} voor elke categorie in een lijst. De lijst bevat duplicates (configs 2×), dus:

ERROR: Encountered two children with the same key, `header-configs`
ERROR: Encountered two children with the same key, `header-session`
ERROR: Encountered two children with the same key, `header-built_in_commands`
ERROR: Encountered two children with the same key, `header-custom_skills`

Gevolg: React kan de VDOM niet correct reconcileren. Het gooit de hele subtree weg en herbouwt deze — een full re-render die alle vorige stappen opnieuw triggert.

Stap 4: Maximum update depth

8× om 08:49-08:50. Dit is React's vangnet dat ingrijpt als een component te diep nestelt in re-renders. Dit gebeurde vermoedelijk toen een useEffect zonder dependency array setState aanriep die weer een re-render triggert die weer... → oneindige loop. React verbreekt de loop na ~50 iteraties.

Stap 5: setState during render (jnM)

8× verspreid over de dag. Een component (jnM in productiecode — mogelijk een minified versie van een chat of header component) roept setState aan terwijl React een andere instantie van hetzelfde component aan het renderen is.

Dit gebeurt vaak kort na i18next promo clusters — wat suggereert dat de render cycle opnieuw start na een herinitialisatie.
